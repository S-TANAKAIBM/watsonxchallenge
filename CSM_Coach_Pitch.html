取り組んでいる課題

IBM CSMはSelect Territory体制で多数の顧客を担当しており、顧客情報が複数システムに分散しているため、状況把握・アクション判断・資料作成に多大な時間を要している。

情報収集・判断コスト
利用状況・NPS・チケット・BP情報・更新日などを毎回手動で収集・整理した上でアクションを決定。担当顧客数が多いほど個別対応の品質が低下する。
チャーンリスクの見逃し
健全性シグナルを定期モニタリングする時間が確保できず、競合比較中・NPS急落・更新接近などの複合シグナルを見落とすリスクがある。
関与者連携の複雑さ
BP・IBM-C・TELは顧客ごとに異なり三者が揃うことはまれ。誰に何を確認・依頼すべきかをCSMが毎回ゼロから判断する必要がある。
QBR・オンボーディング準備工数
高品質な成果物が求められるが、テンプレート化が進んでおらず、CSMごとに品質・工数にばらつきがある。
ターゲットユーザーと相互作用方法

現在のターゲット：IBM CSM（Customer Success Manager）
Select Territory担当・Focus Product 20製品・多顧客を抱えるCSMが顧客接点の前後に使用する。外部システム連携なしで即日利用可能。

将来のターゲット：営業（テリトリー営業・TEL・IBM-C）
顧客のRenewal状況・拡張機会・健全性スコアを営業が把握し、アップセル提案・商談準備に活用。CSMと営業が同一AI基盤で情報を共有することで連携が強化される。

  • IAMトークン取得・エラーハンドリングのデバッグ
  • フロントエンド HTML/CSS/JS の設計と実装
  • PowerShell検証スクリプトの文字コード問題トラブルシュート
  • 複数製品セレクター・参照情報管理UIの実装支援
ICA — プロンプト設計・業務ロジック整理
  • Tech Touch / Human Touch 活動モデルの構造化
  • BP・IBM-C・TEL 関与者別アクション分岐ルールの設計
  • Focus Products（20製品）をプロンプトに明示・制約化
  • 3モード（オンボーディング/NBA/QBR）プロンプト設計・チューニング
  • オンボーディングガイド参照情報ルール策定
  • 思考過程抑制・出力フォーマット標準化

ロードマップ — CSM単独から組織全体へ
Phase 1現在・MVP
CSM向けスタンドアローンWebUIテキスト入力のみで即日利用可能。外部システム連携不要。
3モード・20製品対応オンボーディング / NBA / QBR。Focus Products制約・関与者別分岐。
参照情報管理IBM公式事例・BP実績・URLをUIから登録し全分析に自動反映。
Phase 2短期
営業モード追加アップセル機会検出・商談前ブリーフィング生成・Renewal確率スコアリング。TEL・IBM-C向け出力フォーマット追加。
多製品プロンプト拡張各製品専用のオンボーディングガイドと診断ロジックを整備。製品ごとのベストプラクティスを自動注入。
eNurture連携Tech Touch配信状況をAIが参照しHuman Touch介入判断の精度向上。
Phase 3中期
ISC連携（社内規定対応後）ISC（ibmsc.lightning.force.com）とICN連携し、顧客情報・製品・ライセンス数を自動取得。入力工数をほぼゼロに。
Case Viewer連携Case Viewer（w3.ibm.com/tools/caseviewer）から同製品の過去ケースをwatsonxで検索・自動参照し、類似課題への対応精度を向上。
健全性スコア自動モニタリング全担当顧客のリスクスコアを定期実行しHIGH RISKを自動アラート。CSM・営業共有ダッシュボード。
Phase 4長期展望
全IBMセールス統合CSM・営業・テクニカルセールスが同一AI基盤で顧客360°ビューを共有。提案書・契約ドラフトの自動生成。
社内ナレッジ横展開蓄積された対応履歴・成功事例をwatsonxでインデックス化し、全CSM・営業が検索・活用できる社内AIナレッジベースへ。
